# Environment Variables & Secrets — Specification

## Approach
- **Secrets** (API keys, tokens, passwords) → GitHub Codespace secrets at github.com/settings/codespaces. Available as env vars in the Codespace automatically.
- **Non-secret config** (URLs, feature flags, app IDs) → `.env` files committed in the repo or generated by setup.sh from workspace.json. Read by dotenv.

## How it works today

### Secrets (Codespace secrets)
Set once at github.com/settings/codespaces, then grant access to each workspace repo:
```bash
REPO_ID=$(gh api repos/Org/workspace --jq '.id')
gh api -X PUT /user/codespaces/secrets/SECRET_NAME/repositories/$REPO_ID
```

Current secrets:
- `ANTHROPIC_API_KEY` — for Claude Agent SDK (vibe-ui)
- `WASHMEN_GITHUB_TOKEN` — GitHub PAT for cloning private repos and pushing branches

Future secrets (per project):
- Any secret the project needs (DB passwords, API keys, etc.)
- Set as Codespace secrets, accessed via `process.env.SECRET_NAME`

### Non-secret config (.env files)
Two options:
1. **Committed in repo** — `.env.example` or `.env.development` checked into the project repo. Setup.sh copies or uses as-is.
2. **Generated from workspace.json** — `envFiles` section in workspace.json, written by setup.sh on codespace creation.

### Setup flow
1. Codespace starts → Codespace secrets injected as env vars automatically
2. `setup.sh` runs → clones repos, installs deps
3. `setup.sh` writes `.env` files from workspace.json `envFiles` section
4. Services start → dotenv reads `.env`, Codespace secrets available via `process.env`

## Notes
- AWS Secrets Manager is NOT used — ECS handles that for production, but Codespaces can't use the same IAM role pattern
- Adding a new secret: set it at github.com/settings/codespaces, grant repo access, reference in code via `process.env`
- Non-secret config changes: update workspace.json `envFiles` or the repo's `.env.example`
